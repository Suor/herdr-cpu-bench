# REPORT — slim-fix под нагрузкой (non-idle A/B vs master)

Дата: 2026-06-11/12. Вопрос: может ли событийная (event-driven) детекция из
`slim-fix` — проход на каждый батч PTY-вывода вместо мастеровского тика
300/500 мс — стоить ДОРОЖЕ master в не-idle сценариях?

**Ответ: регрессии нет. Под нагрузкой slim-fix в 2–3 раза дешевле master;
под флудом оба насыщают ядро. Но доля детекции под флудом реальна (37% циклов)
— это резерв оптимизации, не регрессия.**

## Методика

- master (`0722270`) собран из исходников тем же zig-тулчейном, что и slim-fix
  (одинаковый codegen — честный A/B).
- Изолированный bench (отдельная копия `herdr-bench`, приватный сокет/XDG),
  44 панели, 4 idle-агента — конфигурация faithful-harness.
- Пары back-to-back в одной сессии; абсолютные числа дрейфуют с фоновой
  нагрузкой машины, сравнивать только внутри пары.

## Результаты (CPU сервера, замер 12 с)

| Сценарий                                      | master | slim-fix |
|-----------------------------------------------|-------:|---------:|
| Idle, detached (контроль)                     |  5,5%  |   0,42%  |
| Chatty: 4 панели × ~20 строк/с, detached      | 27,1%  | **9,9%** |
| Chatty, attached 190×47                       | 52,8%  | **25,3%**|
| Флуд: `yes` в 1 панели, detached              |  127%  |   122%   |

Флуд ограничен flow-control'ом PTY: `yes` пишет со скоростью потребления,
поэтому обе версии насыщают ~1,2 ядра — это потолок по времени, а не по работе.

## Профили (perf, флат self/children по символам)

- **slim-fix, флуд**: `ghostty_recent_text` (бэкенд `detection_text()`) — 37%
  children; `process_pty_bytes`/`vt_write` — ~27%. Детекция на каждый батч —
  крупнейший потребитель.
- **master, флуд**: `process_pty_bytes` — 57%, `recent_text` — лишь 7,8%
  (детекция тикает 300/500 мс).
- **master, chatty**: `recent_text` — 38% (!) — мастеровский цикл детекции
  безусловно извлекает текст ВСЕХ 44 панелей каждый тик, даже статичных, плюс
  каждый тик форсирует полный рендер. slim-fix извлекает только панели, чья
  сетка менялась — отсюда трёхкратный выигрыш под chatty.

## Выводы

1. Гипотеза «детекция без верхней границы частоты» подтвердилась по факту
   (частота проходов выросла на порядок, под флудом 37% циклов), но не по
   последствиям: всё остальное, что убрал rewrite, перевешивает.
2. Резерв: минимальный интервал между проходами детекции (коалесинг ~100–300 мс
   при непрерывном выводе) срезал бы примерно треть циклов в сценарии
   насыщения. → ветка `proper-fix`.

## Артефакты

- `bin/herdr-master-tmp` — собранный pristine master (без симлинка, для A/B).
- Профили: `/tmp/p-slim-chatty.data`, `/tmp/p-slim-flood.data`,
  `/tmp/p-master-chatty.data`, `/tmp/p-master-flood.data` (могут быть удалены
  перезагрузкой).

# Ветка `proper-fix` (2026-06-12) — устранение найденных резервов

Три коммита поверх `slim-fix` (5 файлов, +257/−6; `bin/herdr-proper-fix` +
симлинк). Все замеры — bench-харнесс, 44 панели, 4 idle-агента, back-to-back.
Четвёртый коммит (пейсинг чтения PTY) вынесен в отдельную ветку `pace-fix` —
см. раздел 4: он связывает скорость скрытой программы с видимостью вкладки,
что признано неприемлемым по UX; держим про запас.

## 1. `cca71f2` — дебаунс детекции (250 мс между output-проходами)

Wake-и от вывода коалесцируются: не больше одного прохода детекции в 250 мс
на панель (худшая добавка к латентности — те же ~порядки, что старый поллинг
master 300/500 мс). Быстрые тики (pending release, восстановление цвета
хост-терминала) дебаунс пропускают.

| Сценарий                      | slim-fix | + дебаунс |
|-------------------------------|---------:|----------:|
| Флуд `yes`, detached          |   122%   |  **97%**  |
| Chatty ×4, detached           |   9,9%   |  **7,1%** |
| Chatty ×4, attached 190×47    |  25,3%   | **10,7%** |

`ghostty_recent_text` исчез из топа профиля флуда; остаток флуда — чистый
ингест PTY (vt_write/memset), неустранимый, пока поток потребляется.

## 2. `fa863ef` — вывод скрытой панели не будит рендер

`PaneTerminal.visible_to_client` (по образцу `hook_authority_present`):
`on_read` ставит `render_dirty`/будит рендер-цикл только для видимой панели.
Грид при этом обрабатывает каждый байт — `pane read`, детекция, статусы
агентов в сайдбаре не затронуты. Сервер пушит видимость после каждого полного
рендера (когда `state.view.pane_infos` отражает экран): видимы панели
активной вьюхи рендер-клиента; фоновые вкладки, другие workspace, detached —
скрыты. Переход «скрыта → видима» планирует один догоняющий рендер (вывод,
успевший между рендером и синком, не теряется).

| Сценарий                          | + дебаунс | + видимость |
|-----------------------------------|----------:|------------:|
| Chatty ×4 (фоновые вкладки), detached | 7,1%  |   **1,7%**  |
| Chatty ×4, attached 190×47        |   10,7%   |   **3,3%**  |
| Флуд `yes` в скрытой панели, detached | 97%   |   **92%**   |
| Болтливая панель В ФОКУСЕ, attached | —       | 20,5% + 4,2% клиент |

Последняя строка — честная цена просмотра занятой панели (полные рендеры
~20/с); уход с её вкладки возвращает 2,9%. Переключение на скрытую болтливую
вкладку моментально даёт живой стрим (проверено e2e), 4/4 агента, доставка
кадров (verify_delivery exit 0).

## 3. `26e880f` — расфокус хост-терминала = «ничего не отображается»

Клиент и так шлёт focus-репортинг хост-терминала (CSI I/O →
FocusGained/FocusLost), сервер хранит его в `ClientConnection.
outer_terminal_focus`. Теперь рендер-клиент считается «отображающим», только
пока его хост-терминал не сообщил о потере фокуса (скрытая вкладка Guake).
Терминалы без focus-репортинга остаются `None` = отображают (консервативно).
Смена фокуса синхронизирует видимость немедленно (layout не меняется, рендер
не нужен); возврат фокуса — это interaction → полный рендер → панели снова
видимы. E2e: стрим 5094 байта/3с → focus-out **0 байт/3с** → focus-in
возобновление 4990 байт/3с.

## 4. `a82bd44` — пейсинг чтения PTY скрытых панелей (ВЫНЕСЕН в ветку `pace-fix`)

**Статус: не в deliverable.** Работает и измерен (см. ниже), но связывает
скорость выполнения скрытой программы с видимостью её вкладки: сборка,
льющая >потолка предупреждений, замедлилась бы при переключении вкладки —
«not pretty» (решение Suor, 2026-06-12). Любая схема дросселирования чтения
лишь отодвигает этот порог, но не убирает связку; единственное решение без
неё — удешевить парсинг невидимого в самом libghostty-vt (см.
`ghostty-vt-memset-issue.md`). Коммит сохранён в ветке `pace-fix` на случай,
если понадобится (например, как защита от runaway-процессов).

Ответ на «can we not consume it immediately?» — технически да. Даже с гейтингом рендера
скрытый флуд стоил ~90% ядра: каждый байт читался и парсился в грид на полной
скорости ради вывода, который никто не видит. Теперь PTY-актор, пока панель
скрыта, читает не больше 8 батчей × 8 КиБ за окно 100 мс (**≈640 КиБ/с ≈
10 тыс. строк/с**). Отложенные данные лежат в кернельном PTY-буфере, и его
бэкпрешер замедляет пишущую программу до скорости потребления — тот же flow
control, что у любого медленного терминала. Флаг видимости расшарен из
`PaneTerminal` (`Arc<AtomicBool>`): фокус на панель снимает пейсинг на
следующей итерации цикла актора.

Тихие скрытые панели не затронуты: бюджет обновляется за окно, первый батч
после паузы читается немедленно, при отсутствии данных актор блокируется
бессрочно (никаких периодических пробуждений — таймерный poll только в
разгар флуда). Запись (ввод), resize и команды не пейсятся. Всё, что ниже
потолка (сборки, хвосты логов, болтливые агенты), не дросселируется вовсе.

| Сценарий (detached)                    | до    | после |
|----------------------------------------|------:|------:|
| Флуд `yes` в скрытой панели (steady)   |  92%  | **6,5%** |
| + 4 chatty панели поверх флуда         |   —   | **8,0%** |
| Флуд В ФОКУСЕ (полная скорость)        | ~119% | ~119% (намеренно) |

Цена: программа, льющая в скрытую панель быстрее 640 КиБ/с, работает медленнее,
пока её не смотрят (для `yes` это и есть смысл; сборки столько не пишут).
Латентность детекции/EOF скрытой панели — до +100 мс. Константы
`HIDDEN_READ_PACE`/`HIDDEN_READ_BURST` в `src/pty/actor/unix.rs` легко крутить.

## Ответы на вопросы фазы 3

1. **Звуковые алармы при скрытом herdr работают.** Детекция агентов не
   зависит от видимости/рендера (она событийная по выводу PTY); StateChanged
   доходит до приложения, звук/уведомления срабатывают. Подавление
   уведомлений по фокусу (`active_tab_suppresses_notifications`) подавляет их
   только когда вкладка АКТИВНА и в фокусе — при скрытом herdr уведомления
   как раз доставляются. Это upstream-поведение, не тронуто.
2. **«Если herdr не потребляет вывод»: полная остановка чтения недопустима,
   а замедленное потребление работает, но отложено (ветка `pace-fix`).** Полная
   остановка заполнила бы кернельный PTY-буфер (порядка десятков КБ) и
   программа зависла бы на write навсегда, пока панель скрыта. Пейсинг же
   ограничивает скорость, не останавливая: программа продолжает работать,
   просто медленнее, если пишет быстрее 640 КиБ/с. Утечек памяти нет в обоих
   случаях: scrollback ограничен `scrollback_limit_bytes` (конфиг, по
   умолчанию ~10 МБ на панель — см. образец конфига в `src/main.rs`).

| Chatty ×4 (фон), detached | 27,1%  |   9,9%   |  **1,7%**  | 1,7%       |
| Chatty ×4 (фон), attached | 52,8%  |  25,3%   |  **3,3%**  | 3,3%       |
| Флуд скрытый, detached    |  127%  |   122%   |   **92%**  | 6,5%       |
| Скрытая вкладка Guake     | = attached | = attached | **= detached (стрим 0)** | — |

Колонка `(+pace-fix)` — отложенная ветка, не deliverable. Остаточные 92%
скрытого флуда на proper-fix — это чистый парсинг потока в libghostty-vt;
правильное лечение — upstream (черновик issue: `ghostty-vt-memset-issue.md`:
38–43% циклов — memset страниц scrollback при recycle, ~1,6 КиБ зануления
на каждую прокрученную строку при полном scrollback).

Тесты proper-fix: 1968/1968 (новый:
`pane_visibility_is_synced_to_runtimes_after_full_render` с фазами фокуса;
тест пейсинга `hidden_pane_reads_are_paced_and_resume_with_visibility` уехал
в `pace-fix`). Clippy: чисто (1 давнее предупреждение в
`src/app/actions.rs:2550`, не наше).
